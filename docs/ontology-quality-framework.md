# 本体质量评价理论框架

**日期**：2026-04-04
**状态**：设计中

---

# 一、目标

定义一套可计算的本体质量评价指标体系，用知识数据反向审计本体设计质量。

**核心原则**：好的本体应该让知识图谱呈现出"结构清晰、边界分明、层次有序"的特征。如果数据呈现出退化信号（超级节点、关系混用、跨层断裂），那是本体设计的问题，不是数据的问题。

---

# 二、理论基础

## 2.1 评价维度

从知识工程理论出发，本体质量可以从 5 个正交维度评价：

```
              ┌──────────────┐
              │  正交性       │  关系类型之间是否语义不重叠
              │ (Orthogonality)│
              └──────┬───────┘
                     │
     ┌───────────────┼───────────────┐
     │               │               │
┌────▼─────┐  ┌──────▼──────┐  ┌────▼─────┐
│ 粒度合理性│  │  层间连通性  │  │ 可发现性  │
│(Granularity)│  │(Cross-Layer)│  │(Discoverability)│
└────┬─────┘  └──────┬──────┘  └────┬─────┘
     │               │               │
     └───────────────┼───────────────┘
                     │
              ┌──────▼───────┐
              │  结构健康度    │  图论指标（联通性、环、传递性）
              │ (Structural)  │
              └──────────────┘
```

## 2.2 每个维度的理论依据

### 粒度合理性（Granularity）

**理论**：本体节点应该在"太泛"和"太细"之间找到平衡。太泛的节点（如"IP"）会与所有东西关联，变成超级节点，失去区分力；太细的节点（如"BGP Keepalive Timer Default Value"）没有足够的关系和证据支撑。

**度量信号**：
- 节点度数分布：应近似幂律分布（少数 hub + 大量低度节点），但不应有极端离群点
- Segment 平均标签数：单个 segment 挂太多本体节点 → 这些节点的概念边界不清
- 节点出现频率：某节点出现在 >80% 的 segment 中 → 太泛

### 正交性（Orthogonality）

**理论**：不同的关系类型应该表达不同的语义。如果两个谓语 P1 和 P2 在节点对上高度共现（凡是有 P1 的地方几乎都有 P2），说明它们的语义重叠——要么合并，要么重新定义边界。

**度量信号**：
- 谓语共现矩阵：对所有节点对 (A,B)，统计同时存在的谓语组合 → 计算两两 Jaccard 相似度
- 谓语分布偏度：如果 54 种关系中 3 种占了 80% 的 facts → 定义不够精细或数据偏差
- 节点内谓语集中度：一个节点的边 90% 都是同一种谓语 → 该节点可能只在一种语境下被提及

### 层间连通性（Cross-Layer Connectivity）

**理论**：五层模型（concept→mechanism→method→condition→scenario）是知识从"是什么"到"在什么场景下怎么用"的完整链路。如果某个 concept 直接跳到 scenario 而中间层缺失，说明知识不完整或本体结构有缺口。

**度量信号**：
- 层间邻接覆盖率：concept→mechanism 有多少对连接？mechanism→method 有多少对？
- 短路率：concept→scenario 直接连接（跳过中间层）的比例
- 断裂节点：某层节点完全没有与相邻层的任何关系
- 层间路径完整度：从 concept 能通过多少条完整路径走到 scenario（经过所有中间层）

### 可发现性（Discoverability）

**理论**：本体节点必须能被 Pipeline 发现。如果一个节点没有别名，alias matching 永远匹配不到它；如果一个关系类型没有对应的正则或 LLM prompt 支持，它永远不会被抽取到。

**度量信号**：
- 别名覆盖率：有 ≥1 个别名的节点比例
- 多语言覆盖率：同时有中英文别名的节点比例
- 关系类型利用率：被 Fact 实际使用的关系类型 / 总定义数
- Pipeline 召回率：节点被 segment_tags 命中的比例 vs 该节点在原始文档中实际出现的比例（需要人工抽检）

### 结构健康度（Structural Health）

**理论**：知识图谱的图论结构应该满足一些基本的健康条件。

**度量信号**：

**联通性**：
- 巨大联通分量占比：去掉 top-N 超级节点后，图是否分裂成有意义的子图？
- 孤立节点比例：零度节点的比例
- 平均最短路径长度：太短说明图过于密集（噪声），太长说明图过于稀疏

**环与传递性**：
- 依赖环检测：`depends_on` 关系是否存在环（A→B→C→A）？环是本体逻辑错误
- 传递归约：如果 A→B→C 且 A→C，是否有冗余？（SUBCLASS_OF 可传递，depends_on 通常不应传递）
- 对称性违反：声明为 symmetric 的关系是否真的双向存在？

**度数分布**：
- Gini 系数：度数分布的不均匀度。太高 → 少数节点垄断；太低 → 图过于均匀（不现实）
- 合理范围：正常知识图谱的 Gini 系数约 0.4-0.7

# 三、指标体系

## 3.1 粒度合理性指标

| 指标 | 计算方式 | 健康范围 | 异常信号 |
|------|----------|----------|----------|
| G1: 度数 Gini 系数 | 节点度数分布的 Gini | 0.4-0.7 | >0.8 超级节点主导 |
| G2: 超级节点比例 | 度 > mean+3σ 的节点占比 | <5% | >10% 粒度太粗 |
| G3: 孤立节点比例 | 零度节点 / 总节点 | <20% | >40% 知识覆盖不足 |
| G4: Segment 平均标签数 | AVG(tags per segment) | 2-5 | >8 节点边界模糊 |
| G5: 节点频率偏度 | 出现在 >80% segments 的节点数 | 0-2 | >5 有万金油节点 |

## 3.2 正交性指标

| 指标 | 计算方式 | 健康范围 | 异常信号 |
|------|----------|----------|----------|
| O1: 谓语共现 Jaccard | 两两谓语在节点对上的 Jaccard 相似度 | <0.3 | >0.5 语义重叠 |
| O2: 谓语分布偏度 | top-3 谓语占比 | <60% | >80% 定义不够精细 |
| O3: 节点谓语集中度 | 单节点最大谓语占比 | <70% | >90% 语境单一 |
| O4: 有效谓语数 | 被 Fact 使用的谓语种数 | >50% 定义数 | <30% 大量废定义 |

## 3.3 层间连通性指标

| 指标 | 计算方式 | 健康范围 | 异常信号 |
|------|----------|----------|----------|
| L1: 层间覆盖率 | 相邻层有连接的节点对比例 | >30% | <10% 层间断裂 |
| L2: 短路率 | concept→scenario 直连 / 总跨层边 | <20% | >50% 中间层缺失 |
| L3: 完整路径数 | concept→mech→method→cond→scenario 完整链路 | >0 | =0 五层模型失效 |
| L4: 层间使用关系类型数 | 跨层 edge 使用了几种 predicate | >5 | <3 关系表达力不足 |

## 3.4 可发现性指标

| 指标 | 计算方式 | 健康范围 | 异常信号 |
|------|----------|----------|----------|
| D1: 别名覆盖率 | 有别名节点 / 总节点 | >80% | <50% 大量节点不可检索 |
| D2: 多语言覆盖率 | 同时有中英别名的节点比例 | >60% | <30% 双语能力不足 |
| D3: 关系类型利用率 | 已使用 / 总定义 | >50% | <30% 定义浪费 |
| D4: 标签命中率 | 被 segment 标记过的节点比例 | >60% | <30% Pipeline 覆盖不足 |

## 3.5 结构健康度指标

| 指标 | 计算方式 | 健康范围 | 异常信号 |
|------|----------|----------|----------|
| S1: 联通分量数 | 去掉 top-5 hub 后的分量数 | 3-10 | >20 图碎片化 / =1 全连通 |
| S2: 依赖环数 | depends_on/requires 图中的环 | 0 | >0 逻辑错误 |
| S3: 传递冗余率 | A→B→C 且 A→C 的比例 | <10% | >30% 抽取噪声 |
| S4: 对称违反数 | 声明 symmetric 但只有单向边 | 0 | >0 数据不完整 |
| S5: 平均最短路径 | 全图 APL | 2-4 | <2 过密 / >6 过稀 |

# 四、从指标到诊断

## 4.1 诊断规则

每个指标异常不是孤立的，而是指向具体的本体问题：

```
G2 超级节点比例过高
  + O2 谓语分布偏度过高
  → 诊断：少数节点 + 少数谓语主导了整个图
  → 建议：检查 top-5 超级节点是否粒度太粗，应拆分

O1 谓语共现 Jaccard 过高 (P1, P2)
  → 诊断：P1 和 P2 语义重叠
  → 建议：合并为一个关系类型，或重新定义语义边界

L1 层间覆盖率低 (concept→mechanism)
  + D1 别名覆盖率低 (mechanism 层)
  → 诊断：mechanism 层不可被 Pipeline 发现
  → 建议：补充 mechanism 层别名，添加种子关系

S2 依赖环数 > 0
  → 诊断：依赖关系定义有逻辑错误
  → 建议：列出环中的节点，人工修正

G4 segment 标签过多
  + G5 万金油节点存在
  → 诊断：某些节点概念太泛
  → 建议：将万金油节点降级为领域标签，或拆分成子概念
```

# 五、实施路径

## 5.1 与现有系统的关系

```
StatsCollector (已有)
  ├── 7 类监控指标 (量化)
  └── 新增: ontology_diagnostics (结构质量)

新增算子:
  ├── ontology_quality (计算全部 5 维 20+ 指标)
  └── 复用 graph_inspect / cross_layer_check / ontology_inspect

Dashboard:
  └── 新增: Ontology Quality 标签页（雷达图 + 问题列表）
```

## 5.2 计算时机

- **不在每次 stats 快照时跑**（太重，涉及全图遍历）
- **单独的定时任务**，每 1 小时或手动触发
- 结果存入 `system_stats_snapshots` 的独立 key

## 5.3 与种子关系的关系

种子关系解决的是 L1（层间覆盖率）和 D3（关系类型利用率）。加种子关系后，这两个指标应该显著改善。质量评价框架可以量化种子关系的效果——加之前 L1=0.08，加之后 L1 应该 >0.3。
