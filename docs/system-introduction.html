系统存在的理由：为什么 RAG 不够

电信领域有一个长期被工具忽视的核心困境：同一个概念在不同厂商、不同标准组织的文档里叫法完全不同，但含义相同或高度重叠。

典型示例
Cisco 文档的 "OSPF neighbor"、Huawei 文档的 "OSPF 邻居关系"、RFC 2328 的 "OSPF adjacency"——三者描述同一件事。
"BGP 路由反射器"（Huawei）= "Route Reflector"（Cisco）= "RR"（缩写）= "边界网关协议路由反射"（部分学术文档）。

网络集成交付工程师做新建网络方案设计或动网割接方案时，必须跨越这道语言鸿沟，在脑子里手动完成术语归一化，才能把从不同厂商文档和标准获取的知识拼在一起。这是一个认知负担极重、非常容易出错的过程。

RAG 的四个结构性缺陷

🔍
缺陷一：无法溯源到具体断言
"BGP depends on TCP"——这个结论来自 RFC 4271、Cisco 技术手册还是 LLM 训练数据？置信度不透明，无法按来源权威性加权。
🕸️
缺陷二：无法做图遍历
"OSPF 宕机会影响哪些服务？"需要沿着依赖边反向追踪。RAG 能找到相关文档，但无法告诉你依赖链。
⚔️
缺陷三：无法检测跨源冲突
两篇文档对同一事实给出矛盾说法，RAG 会把两条内容都返回给 LLM，LLM 可能随机选择其一，用户不会知道存在分歧。
📈
缺陷四：知识不会积累
处理了一千篇文档，RAG 的理解停留在"能检索"层面，不会随文档增加对某知识点产生更高置信度，也不会发现跨文档的知识模式。

本系统的目标是解决这四个问题：可溯源、可遍历、冲突感知、知识积累。

三条设计原则

所有架构决策都是这三条原则的体现。

原则一：知识质量优先于知识数量
宁愿少存一条 Fact，也不存一条低质量的 Fact。一条错误的 "A depends_on B" 关系在故障扩散分析时会给出错误影响面，比没有数据更危险。
原则二：真相来源只有一个，其他都是投影
YAML 文件是本体的唯一真相。Neo4j 和 PostgreSQL 里的本体数据都是从 YAML 派生的运行时投影，可以随时用 scripts/load_ontology.py 重建。这是整个系统最重要的不变量。
原则三：每个阈值和权重都有领域依据
置信度公式的权重、SimHash 阈值、候选词门控阈值——都不是随意选择的数字，背后都有针对电信文档特征的分析。如果要修改，先理解为什么是这个数字。
3
七层架构

为什么算子层和管线层必须分开

这是系统中最重要的架构切割线。Pipeline 处理的是文档加工：有状态、有序、单向，由 Worker 轮询触发，不接受 HTTP 请求。Operator 处理的是知识查询与操作：无状态，接受 HTTP 请求。分开后两层独立演化——加新算子不碰 Pipeline，改 Stage 不影响 API。

唯一组装入口 app_factory.py

build_app() 是系统的依赖注入根（Composition Root）。所有 Provider 实例（PostgresRelationalStore、Neo4jGraphStore、LLMProvider…）在这里被实例化，然后注入到 SemanticApp 单例。开发模式只需一个 build_dev_app()，把所有 Provider 替换成 Fake 实现，业务代码一行不改。

五层本体模型

本系统的知识模型使用五层结构。这五层来自对电信集成交付知识结构的实际分析：一个有经验的交付工程师在做网络设计和割接方案时，脑子里同时维护这五类知识——缺少任何一类都会导致方案不完整。

①
Concept — 概念层
Neo4j: OntologyNode  |  YAML: domains/ip_network.yaml  |  当前节点数: 114
YANG 级别可配置对象——接口、协议实例、策略、VPN 实例等。回答"是什么"。例：IP.BGP、IP.OSPF、IP.VRF。节点 ID 遵循 {DOMAIN}.{NAME} 规范。
②
Mechanism — 机制层
Neo4j: MechanismNode  |  当前节点数: 24
协议算法和转发机制——BGP 建立邻居的 OPEN/KEEPALIVE 消息交换、OSPF 的 SPF 算法、TCP 三次握手。回答"背后怎么工作的"。
③
Method — 方法层
Neo4j: MethodNode  |  当前节点数: 22
配置和排障操作步骤——neighbor x.x.x.x remote-as 命令序列、BGP 邻居重置流程、OSPF 邻接关系建立步骤。回答"怎么操作"。
④
Condition — 条件层
Neo4j: ConditionRuleNode  |  当前节点数: 20
适用条件、约束和决策规则——Hold Timer 超时导致 BGP 邻居断开、MTU 不匹配阻止 OSPF 邻接。回答"在什么条件/约束下"。
⑤
Scenario — 场景层
Neo4j: ScenarioPatternNode  |  当前节点数: 13
典型部署模式和业务场景——多 AS 边界路由黑洞、5G N6 接口 VRF 隔离方案。回答"什么业务场景下用，会引发什么后果"。
为什么不用扁平本体（OWL 式）
扁平本体能回答"是什么"，但无法回答"在什么情况下用"和"为什么失败"。五层结构对应五种独立的认知框架：集成交付工程师关心方法和场景，方案设计师关心条件和机制，架构师关心概念关系。每层对应一种 Neo4j 节点标签，MATCH (n:MechanismNode) 不扫描其他层节点，查询性能与总节点量解耦。

Neo4j — 图数据库
原生图存储 · 图遍历
  • 本体推理层：OntologyNode 间关系边
  • 知识溯源层：Fact→Evidence→Segment→Document
  • 5 种节点标签 × 77 种关系类型
  • 依赖闭包：O(1) 邻居遍历 vs PostgreSQL O(logN)
  • 动态关系类型支持（LLM 发现新谓语）
📦
MinIO — 对象存储
原始 HTML + 清洗文本
  • raw/ — 原始 HTML（SHA-256 去重）
  • cleaned/ — Pipeline Stage 1 清洗文本
  • 内容去重：相同 SHA-256 只存一份
  • 单文档可达 1MB+（如完整 RFC）
  • pgvector 提供语义向量检索（HNSW 索引）
为什么用 pgvector 而不是独立向量数据库
本系统的语义搜索从来不是"在所有向量里找最近的"，而是"在 lifecycle_state='active' 且 segment_type IN ('definition', 'config') 的段落里找最近的"。这类混合查询在 pgvector 里是一个 WHERE 子句；在 Pinecone/Weaviate 里是向量搜索加后过滤，实现复杂且性能不稳定。数量级在百万段落以内，pgvector HNSW 索引完全够用，且零额外运维成本。

为什么 Fact 不直接连接 OntologyNode（无 ABOUT_SUBJECT/ABOUT_OBJECT 边）：

本体图保持干净
OntologyNode 之间只有语义关系边，是多条 Fact 蒸馏后的聚合结论。图遍历直接走本体边，不被 Fact 节点打断路径。
聚合语义更准确
3 条 Fact 说同一件事，在本体图上只产生 1 条边（fact_count=3, confidence=max），图不退化为 Fact 图。
生命周期独立
Fact 经历 active→conflicted→superseded→merged 状态流转，完全在 PostgreSQL 侧管理，Neo4j 本体图不受影响。

7 阶段处理管线

Pipeline 由 Worker 持续轮询 documents 表中 status='raw' 的记录触发。documents.status 字段精确记录每篇文档卡在哪里，故障诊断从分钟级变成秒级。阶段可以独立重跑，不需要重新下载清洗所有文档。

Stage 1
Ingest — 文档摄取与清洗
HTTP 抓取 → HTML → 结构化文本转换（h1/h2/h3 转 Markdown # 标记保留层级）→ 句子密度质量门控 → SHA-256 内容去重 → doc_type 检测（spec/guide/vendor）→ 存 MinIO，写 documents(status='cleaned')
Stage 2
Segment — 分段与篇章关系标注
4 步联合分段：① 结构切分（按 Markdown 标题和换行）→ ② 段落级 LLM 批量分类（17 种 segment_type：definition/mechanism/config/procedure/constraint/example…）→ ③ 合并相邻同类型段落（同 section，≤1024 tokens）→ ④ 长度控制（滑动窗口 512/64 兜底）。所有相邻段落对发送 LLM 进行 RST 篇章关系分类（批量 15 对，含 segment_type 提示），结果存 t_rst_relation（带 nuclearity）。
Stage 3
Align — 本体对齐
5 层对齐策略：① 词边界精确匹配 → ② 别名映射（828 条）→ ③ Embedding 模糊匹配（cosine > 0.80）→ ④ 候选词发现（LLM 分类：new_concept/variant/noise）→ ⑤ RST 传播（零命中段落从 RST 邻居借取本体标签，按关系类型加权 0.30–0.85）。对齐结果存 segment_tags。
Stage 3b
Evolve — 候选词轻量重评分
每次 Pipeline 循环执行（不等 Maintenance）。对所有 status='discovered' 候选词，基于最新累积 source_count 重新计算 6 维评分。score ≥ 0.85 且通过 6 道门控 → 标记 auto_promote。单文档视野不足以判断是否入库，批量视野才行。
Stage 4
Extract — 五级优先三元组抽取
五级优先：P0 RST 结构推断（Constraint→constrained_by、Prerequisite→requires…无需 LLM）→ P1 单段 LLM 抽取（≥3 锚点 + 引用验证 80% 重叠）→ P2 合并上下文 LLM（Elaboration/Sequence/Causation/Evidence/Background/Exemplification 触发合并）→ P2-chain 多跳 RST 链（3–4 段连续关系）→ P3 双节点共现兜底。结果存 facts + evidence。
Stage 5
Dedup — 去重与冲突检测
SimHash 结构去重 + Embedding 语义去重（cosine > 0.90）→ Fact 合并。本体驱动冲突检测：D4 仅对 cardinality: one 函数型谓语触发（S≈S, O≠O, P 相同），非函数型谓语跳过；语义冲突检测（S≈S, O≈O, P 不同）→ 写 governance.conflict_records，通知人工仲裁。
Stage 6
Index — 入图 + 向量索引
置信度门控（segment ≥ 0.5，fact ≥ 0.5）→ 写入 Neo4j 本体推理层（聚合边）和知识溯源层（Fact→Evidence→Segment→Document 链）→ 写入 pgvector 向量索引 → documents.status='indexed'。
为什么 Stage 4 LLM 是硬要求，共现兜底不降级
共现关系只能表达"A 和 B 在同一段文字里出现了"，谓语永远是 co_occurs_with，无方向性，无语义类型，无法支撑依赖闭包、影响传播、故障推理——这些是整个系统的核心价值。允许系统以共现兜底继续跑，会导致大量低质量关系进入 Neo4j，且工程师会误以为系统在正常工作。停摆等 LLM 恢复的代价远低于数据清理代价。
8
段落级篇章关系（RST 适配）

本系统借鉴 RST（Rhetorical Structure Theory）的框架——"话语单元之间存在修辞功能关系"——但在段落粒度重新定义了类型边界，是一套领域适配的段落级篇章关系词表，不是标准 RST 的直接使用。

标准 RST 有 21 种子句级关系，在段落粒度强行区分 Elaboration/Explanation/Restatement/Summary 会导致 LLM 分类随机漂移。本系统保留 10 种在段落粒度有明确语言学信号可区分的类型。

关系类型 Nuclearity 典型语言信号 在 Stage 4 的用途
Elaboration NS 无显式连接词，后段展开前段细节 触发合并上下文重试；RST 传播权重 0.85
Sequence NN first / then / next / finally 触发合并；RST 链式抽取（≥3段≤4跳）
Causation NS because / therefore / as a result 直接推导谓语 triggers；触发合并
Constraint SN MUST / MUST NOT / requires / shall 直接推导谓语 constrained_by；不触发合并
Prerequisite SN before / in order to / provided that 直接推导谓语 requires
Condition SN if / when / in the case of 直接推导谓语 applies_when
Exemplification NS for example / such as / e.g. 直接推导谓语 has_example；触发合并
Evidence SN according to / as shown in / RFC XXXX states 触发合并；RST 传播权重 0.60
Background SN historically / originally / in the context of 触发合并；RST 传播权重 0.60
Contrast NN however / on the other hand / while / whereas 不触发合并；RST 传播权重 0.30（最低）

Nuclearity — 核心性

每条 RST 关系带有 nuclearity 属性（NS/SN/NN），表示哪一方是话语的核心（承载主要信息量）：

  • NSsrc 段是 Nucleus，dst 段是 Satellite → src 是主体，dst 提供辅助信息
  • SNdst 段是 Nucleus，src 段是 Satellite → dst 是主体
  • NN两段地位相等

在 Stage 4 RST 直接推导 Fact 时，nuclearity 决定 subject/object 赋值——NS 时 src 为主语，SN 时 dst 为主语，防止关系方向颠倒。

RST 在 Stage 3 的另一用途：传播对齐

对齐后仍然零命中本体节点的"孤立"段落，从 RST 邻居段落借取本体标签，权重由关系类型决定（Elaboration/Sequence: 0.85 → Contrast: 0.30），核心性节点额外加 0.05 权重。

9
置信度评分公式
confidence = 0.30 × source_authority
           + 0.20 × extraction_method
           + 0.20 × ontology_fit
           + 0.20 × cross_source_consistency
           + 0.10 × temporal_validity
0.30
source_authority
在电信领域，谁说的比怎么提取更重要。IETF RFC 经过严格标准化和多轮同行评审，错误率极低；论坛帖子的错误率远高于此。
0.20
extraction_method
rule（0.85）高于 llm（0.70）——精心为电信领域设计的规则对特定模式准确率极稳定；LLM 在边界案例上不确定性更高。
0.20
ontology_fit
Fact 的主体/客体是否锚定在本体节点上。锚定越精确，知识越可复用，图遍历结果越可信。
0.20
cross_source_consistency
多个来源验证同一事实时，置信度自然累积。从 1 个来源到 3 个来源的提升最显著，之后边际递减。
0.10
temporal_validity
电信标准生命周期非常长（RFC 793/TCP 发布于 1981 年至今有效）。与 Web 内容不同，陈旧风险远低于其他领域，权重最低。

候选词治理：三层过滤
Stage 3（每篇文档）          粗筛：LLM 分类 + 停用词 + 初始评分
      ↓
Stage 3b（每次 Pipeline 循环）  轻量重评分：基于累积 source_count 更新分数
      ↓
Maintenance（每天 03:00 CST）   深度精筛：Embedding 聚类 + LLM 批量分类
      ↓
Human Review（随时）           终审：approve / reject / merge

为什么 Stage 3 不直接决定是否入库：Stage 3 的视野只有当前这篇文档。"BGP-LS"第一次出现时 source_count=1，分数低。当第 3 篇文档也提到它之后，source_count=3，分数超过 0.70，触发自动晋升。这个逻辑需要全局数据，不能在单文档的 Stage 3 里判断。

Maintenance 在 03:00 CST 的原因：凌晨 3 点是爬虫和 Pipeline 的低谷期；03:00 CST = 19:00 UTC，也是欧美技术人员工作时间结束后，避免与外部 LLM API 高峰期重叠。

Variant 合并不丢知识：候选词被判定为已有本体节点的变体时，不是简单删除——它出现过的段落通过别名扩展被关联到正确节点（alias_map 增加别名，BackfillWorker 回填已有段落的 segment_tags）。知识归位，不丢失。

11
LLM 集成设计

为什么使用闭合词表（77 种关系类型）

LLM 在没有约束时会发明谓语：is_affected_by、impacts_performance_of、can_be_configured_via… 谓语词表开放会导致 Neo4j 里出现数百种关系类型，图遍历变得不可控。

例外机制：当现有谓语都无法准确表达文本中的关系时，LLM 被允许创造新谓语。这些新谓语以 candidate_type='relation' 进入候选词池等待人工审核，不直接入本体。这是"开放性 + 可控性"的平衡。

nodeContext 注入

Stage 4 在调用 LLM 前，先从 segment_tags 加载这个段落关联的本体节点，构成 nodeContext 放进提示词。LLM 有"线索"时抽取质量更高——告诉 LLM"这段文本里出现了 BGP 和 TCP"，LLM 就不需要从零识别实体，可以集中精力判断它们之间的关系类型。

为什么只让 subject/object 使用本体节点 ID

LLM 提示词要求 subject 和 object 必须是提供的节点 ID 列表中的 ID，不接受自由文本实体。如果允许 LLM 自由命名实体（"BGP router"、"Border Gateway Protocol Router"、"bgp-router"——可能是同一个东西），Neo4j 里会出现大量游离节点，图会迅速变得混乱，图遍历结果不可信。强制节点 ID 保证了所有 Fact 都锚定在本体中。

Circuit Breaker — 阻塞式熔断器

LLMExtractor 连续失败 3 次后触发熔断。熔断后不降级运行——所有依赖 LLM 的线程阻塞等待恢复（每 30 秒轮询 LLM 可用性），同时执行已注册的 keep-alive 回调（数据库心跳等）防止连接池断开。HTTP 4xx（429 除外）不计入熔断阈值，429 指数退避重试，5xx 计入失败。LLM 恢复后自动解除阻塞，从断点继续处理。

RST 批量分类：批次大小 15

一篇完整 RFC 可能有 246 个相邻段落对。所有对放在一个 LLM 调用会产生约 200K 字符的 prompt，导致超时。每对关系完全独立（A→B 只依赖这两个段落），可以安全地分批处理，批次大小固定为 15 对，保证 prompt 在合理大小内。

为什么选 bge-m3

三级回退策略

if http_bge_m3_available():
    return HttpEmbeddingClient()        # WSL2 systemd 服务，:8000，推理最快
elif ollama_is_available():
    return OllamaEmbeddingClient()      # 无 Python 依赖，模型由 Ollama 管理
else:
    return SentenceTransformerClient()  # pip install，首次加载慢，纯 Python 兜底

三者的输出格式完全相同（1024 维 float list），上层代码不感知差异。HTTP BGE-M3 通过 EMBEDDING_HTTP_URL 配置，Ollama 通过 OLLAMA_URL + OLLAMA_EMBED_MODEL 配置。

Worker 进程：4 线程协作模型
主进程 (worker.py)
├── _crawler_thread     — 60+ 种子 URL（RFC Editor / Huawei / Juniper / Arista / FRRouting 等 10 站）
│                         持续爬取 pending 任务，写 raw 文档到 MinIO
├── _pipeline_thread    — LLM 硬依赖（熔断时所有线程阻塞等待恢复，不降级运行）
│                         持续处理 raw 文档，走 7 阶段 Pipeline
├── _stats_thread       — 每 5 分钟采集系统指标快照
└── _maintenance_thread — 每天 03:00 CST 执行本体维护（候选词归类 + Embedding 去重）

线程间协作：数据库状态而不是共享内存

crawler_thread  写  documents(status='raw')
                           ↓
pipeline_thread 读  status='raw' 并处理
                           ↓
pipeline_thread 写  evolution_candidates
                           ↓
maintenance_thread 读  evolution_candidates 并归类

线程之间没有直接通信（无 Queue、无 Event 信号），完全通过数据库状态间接协调。任何一个线程崩溃和重启都不影响其他线程；重启后从数据库当前状态续跑，不丢失进度。

为什么使用 threading 而不是 asyncio 或 multiprocessing：Pipeline 处理的瓶颈是 I/O（LLM API 调用 5–30 秒、MinIO 读写、Neo4j 查询），不是 CPU。Python GIL 对 I/O 密集型线程没有实质限制。asyncio 需要改写成响应式风格，且 psycopg2 不天然支持；multiprocessing 进程间通信 overhead 更大，不必要。

semcore 框架抽象层

semcore 是零依赖的 ABC 层（Python 标准库 ABC，无任何第三方包）。它存在的核心原因：

让 Pipeline Stage 依赖接口而不是实现

没有 semcore 时，Stage 代码直接引用 psycopg2.connect() 或 neo4j.GraphDatabase.driver()，导致单元测试必须启动真实数据库，开发模式需要所有外部服务。

有了 semcore 的 RelationalStore ABC，Stage 代码只依赖接口，开发模式注入 FakePostgres（SQLite :memory:）和 FakeNeo4j（Python dict），单元测试零外部依赖。

semcore/semcore/
├── abc/
│   ├── stage.py           — Stage ABC (process, _run, retry)
│   ├── operator.py        — SemanticOperator ABC
│   ├── relational_store.py — RelationalStore ABC
│   ├── graph_store.py     — GraphStore ABC
│   └── object_store.py    — ObjectStore ABC
src/
├── providers/             — 生产实现 (PostgresRelationalStore, Neo4jGraphStore...)
└── dev/                   — 开发实现 (FakePostgres, FakeNeo4j, FakeMinIO)

开发模式（无任何外部服务）

python run_dev.py 启动开发模式：SQLite :memory:（两个知识库和爬虫库的 in-memory 替代）、Python dict（Neo4j 替代）、本地文件（MinIO 替代）。所有 Pipeline Stage 和算子代码一行不改即可在开发模式下运行。

次要原因：semcore 未来可以独立发布为 pip 包，供其他领域的语义知识库项目复用框架部分，不引入任何电信领域业务代码。
